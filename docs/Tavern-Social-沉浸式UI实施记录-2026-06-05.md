# Tavern Social 沉浸式 UI 实施记录

日期：2026-06-05  
项目目录：`D:\tavern_helper_template-main`

本文档用于在聊天记录丢失后恢复本次工作的完整上下文。它只记录独立应用
`Tavern Social` 的沉浸式 UI 重构，不涉及旧的酒馆内小手机脚本。

## 一、目标

1. Windows 使用联系人与内容区双栏布局。
2. Codex 窄屏和手机使用五项底部导航。
3. 设置从主界面移出：桌面为弹窗，手机为独立页面。
4. 增加最近会话排序、真实未读和打开会话后已读。
5. 默认跟随系统浅色或深色主题。
6. 当前角色头像决定界面强调色；头像无法采样时使用稳定回退色。
7. 保持原有私聊、主动消息、关系、动态、事件、备份和角色卡能力兼容。

## 二、数据层改动

### 2.1 会话已读时间

`ConversationProfile` 新增：

```ts
lastReadAt: number;
```

迁移规则：

- 旧会话没有 `lastReadAt` 时，使用原会话的 `updatedAt`。
- 这样升级后不会把全部历史消息错误显示为未读。
- 新建会话的 `lastReadAt` 使用新建时刻。

### 2.2 未读规则

- 只统计 `lastReadAt` 之后的角色、模型和主动消息。
- 用户自己发送的消息不计未读。
- 打开角色私聊时更新 `lastReadAt`。
- 当前私聊在前台收到模型回复时直接视为已读。
- 后台主动消息继续保持未读，直到用户打开对应私聊。

新增状态方法：

- `conversationFor(characterId)`
- `unreadCountFor(characterId)`
- `markConversationRead(characterId, readAt)`

## 三、桌面 UI

桌面阈值为宽度大于 `980px`。

左栏固定约 `280px`，包含：

- 品牌与设置入口
- 当前世界切换
- 联系人搜索
- 角色卡导入
- 按最后消息时间排序的最近联系人
- 头像、最后消息、时间和未读标记

右侧内容区包含：

- 私聊
- 动态
- 事件

聊天头部只显示角色头像、名称和关系摘要。主动调度参数不再出现在聊天主界面。

桌面设置使用居中弹窗，左侧固定显示六个分类：

1. 世界与角色
2. 模型连接
3. 关系状态
4. 主动消息
5. 通知
6. 数据与运行

## 四、Codex 窄屏与手机 UI

宽度小于或等于 `980px` 时使用移动结构，不再把通讯录堆在聊天上方。

固定底部导航：

1. 消息
2. 联系人
3. 动态
4. 事件
5. 设置

页面行为：

- “消息”显示按最后消息时间排序的最近会话。
- “联系人”显示当前世界角色、世界切换、搜索和角色卡导入。
- 点击会话或联系人后进入全屏聊天。
- 全屏聊天显示返回按钮并隐藏底部导航。
- “动态”和“事件”各自使用独立页面。
- “设置”先显示分类首页。
- 点击设置分类后进入详情页，同时隐藏底部导航。
- 设置详情页使用顶部返回按钮回到设置首页。

## 五、主题与角色色

主题通过 CSS 变量实现：

- 默认浅色变量。
- `prefers-color-scheme: dark` 下启用深色变量。
- 卡片、边线、输入框、遮罩和阴影都使用主题变量。

角色强调色：

1. 先根据角色 ID 生成稳定的柔和回退色。
2. 角色头像加载成功后，通过画布采样缩略图平均色。
3. 对采样结果限制饱和度和亮度，避免按钮及文字失去对比度。
4. 外部头像跨域失败、图片损坏或无头像时继续使用稳定回退色。
5. 没有当前角色时使用默认靛蓝色。

强调色用于：

- 自己发送的消息气泡
- 主要按钮
- 当前选中项
- 未读圆点和数字
- 当前头像的细微高亮

## 六、空状态

- 配置模型

导入角色后但尚未聊天时，显示当前角色专属的对话引导。

## 十二、角色卡开场消息生成规则

追加日期：2026-06-06

### 12.1 新规则

- 导入 JSON 或 PNG 角色卡时，不再把卡内 `first_mes` 或 `first_message`
  直接写入私聊。
- 卡内原开场白不参与新开场消息的模型提示。
- 新开场只使用角色描述、性格、当前场景、关系状态、世界书和近期世界事件。
- 首次私聊自动调用当前模型生成一条全新的开场消息。
- 新开场限制为聊天软件中的自然私信，控制在 1 到 4 个短段落内。
- 禁止生成长篇小说式开场，禁止替用户行动或说话。
- 没有配置模型时保持空会话，并提示用户配置模型；保存模型配置后自动重试。
- 生成失败时不写入占位消息，再次打开空会话会重新尝试。

### 12.2 旧数据迁移

- 旧版本中来源为 `imported_first_message` 的消息会在加载状态时移除。
- 用户消息、模型回复、主动消息和新生成的开场消息不受影响。
- 新生成开场消息的来源标记为 `generated_opening`。

### 12.3 加载优化

- PNG 角色卡解析器改为仅在实际导入 PNG 时动态加载。
- JSON 角色卡导入和普通应用启动不再提前加载 PNG 解析模块。

## 二十一、角色手机动态、评论与自动发布

追加日期：2026-06-06

### 21.1 动态定义修订

角色动态只表示角色本人拿着手机发布到朋友圈、社交平台或个人主页上的内容。

- 提示词以“你就是这个角色”开始。
- 明确角色正在使用自己的手机。
- 强制使用角色本人的第一人称口吻。
- 不再生成旁观者记录、“她正在做什么”或系统状态播报。
- 不把动态写成对用户的私聊回复。
- 只输出角色真正发布的正文。
- 继续提供设备当前日期、星期、时分和时段，保证发帖内容符合生活作息。
- 继续提供最近动态，避免连续重复相同文案。

### 21.2 移除发布身份

动态页删除“发布身份”下拉菜单。

- 页面输入框固定用于用户发布自己的动态。
- 独立按钮“让当前角色发动态”用于角色生成发帖。
- 当前角色来自消息或联系人页面中的已选角色。
- 角色按钮会直接使用角色手机发帖提示词，不再让用户在动态编辑器中切换身份。

### 21.3 评论系统

每条动态新增评论区：

- 用户可以评论任意角色动态。
- 评论按动态独立保存。
- 用户自己的动态显示“让当前角色评论”按钮。
- 角色评论提示词要求角色本人在手机评论区自然打字。
- 评论参考动态正文、发布者、关系状态、角色性格、最近聊天和已有评论。
- 禁止动作、括号旁白、心理描写和“评论：”标题。
- 评论生成中、成功与失败信息直接显示在动态页内。

兼容接口若返回分段内容数组、嵌套文本或推理文本，模型解析器会尝试提取正文。若评论请求连续两次返回空正文，则根据当前关系阶段写入一条简短保底评论，避免按钮无结果。

### 21.4 自动动态

每个角色新增独立 `AutoMomentSchedule`：

- `enabled`
- `baseIntervalMin`
- `baseIntervalMax`
- `quietHours`
- `dailyLimit`
- `nextAttemptAt`
- `lastPostedAt`
- `statusReason`

默认规则：

- 自动动态默认启用。
- 随机间隔为 `4-10 小时`。
- 每日上限为 `2` 条。
- 安静时段为 `00:00-07:00`。

自动动态与主动私聊共用应用运行期和原生后台唤醒，但各自独立判断时间、每日上限和生成内容。

### 21.5 不补发规则

- 应用运行期间到达时间才尝试生成。
- Windows 托盘驻留时可继续检查。
- Android 仅在系统允许后台任务时检查。
- 应用关闭或后台任务错过的自动动态不补算、不补发。
- 下次应用运行时仅重新安排未来时间。
- 无网络、无模型、预算不足、安静时段或每日上限时，本次延后或跳过。

### 21.6 数据迁移

- 旧动态自动补充空评论数组。
- 旧手动动态继续视为用户动态。
- 旧角色自动补充默认自动动态计划。
- 重新导入同名角色卡时保留已有自动动态计划。

## 二十、动态时间切片提示词

追加日期：2026-06-06

### 20.1 动态定义

角色“动态”不再只理解为朋友圈文案，而是角色在当前时间切片中的生活记录。每次生成只选择一种形式：

1. 手机动态：角色刚刚在朋友圈、社交平台或个人状态里实际发布的内容。
2. 当前状态：角色没有发手机动态，记录她此刻正在做的事情。

角色不是每个时间段都会发手机动态。系统应先判断当前情境，再决定更合理的形式。

### 20.2 时间上下文

提示词会在每次生成时读取设备当前时间，并明确提供：

- 年、月、日。
- 星期。
- 小时和分钟。
- 清晨、上午、中午、下午、晚上或深夜时段。

模型需要根据角色身份、年龄、职业或学业、性格、作息、当前世界、近期事件、关系状态和最近聊天判断此刻的合理生活情境。

### 20.3 生成约束

- 输出必须符合当前日期、星期和时段。
- 上课、上班、睡觉、通勤、吃饭、休息和娱乐等安排应合理。
- 最近聊天只作为生活背景，不能把动态写成对用户的私聊回复。
- 不默认角色正在等待用户。
- 没有设定或事件支持时，不凭空确定天气、地点、节日、重大事件或用户互动。
- 手机动态使用角色本人的自然口吻。
- 当前状态直接描述角色正在做的具体事情。
- 不输出 A/B 标记、“动态：”或“当前状态：”标题。
- 不使用括号动作、舞台说明、心理旁白、小说段落或聊天气泡标签。
- 通常只输出一到三句。

### 20.4 连续性

提示词会加入该角色最近三条动态，要求模型避免重复同一件事或近似措辞，使角色生活具有时间连续性。

### 14.1 网络聊天输出预设

所有普通回复、首次开场和主动消息统一加入网络聊天预设：

- 输出应像真实即时通讯，而不是小说、书信、客服回答或完整作文。

模型格式：

```text
<msg>第一条消息</msg>
<msg>第二条消息</msg>
<sticker:表情包名称>
```

应用会把每个 `<msg>` 渲染成独立气泡。旧模型没有输出标签时，应用仍会按自然段和
