# PalTavern RP 工作台产品整理

## 背景

本轮讨论的方向不是立即替换现有 UI，而是记录一个后续产品方向：让 PalTavern 的“世界”从设置项升级为长 RP 的工作台。当前实现应继续以原项目 UI 为参考，后续若重构，必须先保留原有交互、文案和页面感觉，再逐步迁移逻辑。

## 产品目标

RP 工作台要服务长期角色扮演，而不是把事件、时间线、设定和角色管理全部堆到一个大页面里。它应该让用户快速知道当前世界发生到哪里、接下来可以和谁互动、有什么事件需要处理，以及哪些内容已经成为长期记忆。

## 导航方向

建议的主导航为：

- 消息
- 角色
- 世界
- 动态
- 设置

事件不再单独占主入口，由“世界”承接。时间线继续作为长期记忆系统存在，但在 UI 上收进世界详情或记忆详情里，不再抢主导航位置。

## 世界工作台的核心内容

世界入口第一屏应围绕“当前场景续写”组织，而不是围绕设置表单组织。第一版可包含：

- 当前世界名称
- 当前地点
- 当前剧情状态
- 当前目标
- 当前场景摘要
- 当前世界角色
- 进行中的事件
- 最近记忆摘要

这些内容是为了帮助用户继续 RP，而不是成为一个复杂的管理后台。

## 事件节拍

事件适合变成世界里的剧情节拍：

- 用户可以手动创建事件。
- AI 可以根据当前世界、角色和场景生成事件草稿。
- 定时事件可以复用主动消息的节奏模型，但应由用户在创建世界时选择是否开启。
- 事件完整内容在世界工作台显示。
- 相关角色聊天里只出现轻提示，不应污染所有聊天。
- 用户处理事件后，可以点击结束事件。
- 结束事件后，结果应归档为世界记忆。

## 时间线与记忆

时间线的底层逻辑不应被轻易重写。它仍然负责保存聊天、动态、事件、关系变化、主动消息、今日简报、角色状态和用户手动记录。UI 上可以改称“记忆”或“世界记录”，放进世界详情中。

原则：

- 记忆系统保留。
- 入口收起来。
- 只展示摘要时，不删除完整记录。
- 能进入模型上下文的记录仍然需要明确标记。

## 多世界原则

多个世界之间应保持隔离：

- 每个世界有自己的角色列表。
- 同名角色在不同世界中也应视为独立角色。
- 聊天、动态、事件、记忆、关系状态都按世界隔离。
- 第一版世界管理只做创建、切换和编辑，不做删除，避免误删完整世界数据。

## 当前实现约束

在重新整理源码时，当前原项目 UI 是参考基准。整理代码时应遵守：

- 不改现有 UI 文案。
- 不改现有页面结构和交互含义。
- 不为了新工作台提前替换旧页面。
- 优先整理源码目录、模块边界和依赖关系。
- 任何后续 UI 重构都必须先做单独设计，再基于原 UI 迁移。
