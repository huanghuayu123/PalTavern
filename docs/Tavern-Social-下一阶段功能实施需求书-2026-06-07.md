# Tavern Social 下一阶段功能实施需求书 v0.1

日期：2026-06-07

本文档用于指导 Tavern Social 下一阶段功能开发。它承接当前已有的私聊、动态、事件、主动消息、关系状态和备份能力，目标是把应用推进为一个可长期运行、可积累世界记忆、可解释且可回滚的角色社交世界。

## 1. 项目目标

Tavern Social 下一阶段的目标，是把当前“角色私聊、动态、事件、主动消息”的能力，推进成一个可以长期运行、持续积累关系和生活痕迹的角色社交世界。

成功后的体验不是“角色在聊天窗口等用户输入”，而是“角色在同一个世界里生活、记住、反应、错过、靠近或疏远”。用户能知道最近发生了什么、角色为什么来找自己、某段剧情怎样改变了关系，以及当模型输出不合适时如何撤回影响。

## 2. 当前基础能力

当前已具备：角色卡导入、独立私聊、关系状态、主动消息、动态与评论、世界事件、写卡工具、表情包、模型配置、本地备份、桌面与移动构建。

后续新增功能应复用现有世界、角色、消息、动态、事件、关系和备份结构，不另建一套平行系统。

## 3. 下一阶段总设计原则

- 世界连续性优先：聊天、动态、事件、主动消息和关系变化应沉淀为可回顾、可引用的世界记录。
- 可解释而不过度暴露：告诉用户“为什么发生”，不展示提示词或技术日志。
- 长期档安全：高影响自动行为必须可追溯、可撤销或可修正。
- 角色像人在生活：不要让所有互动都围绕 user，也不要让角色像定时任务。
- 少打扰但有存在感：自动动态、主动消息、简报和角色间互动需要频率保护。
- 跨世界隔离：时间线、状态、简报、关系和互动都只属于当前世界。
- 备份兼容：新增数据进入完整备份；旧数据恢复时自动补默认值。
- UI 安静成熟：新入口像通讯工具的一部分，不做复杂后台感界面。

## 4. 功能一：世界时间线 / 共同记忆

世界时间线是下一阶段的数据底座，用来记录“这个世界发生了什么”。它聚合私聊重点、动态、评论、事件、关系变化、主动消息和系统简报。

### 核心交互

- 新增“时间线”入口，按时间倒序展示当前世界记录。
- 条目默认展示短摘要，可展开查看详情。
- 用户可以手动添加剧情备注。
- 普通聊天不自动全量进入时间线，只允许手动标记重要消息或由高价值事件自动写入。

### 记录来源

- 主动消息：角色主动联系时写入。
- 动态与评论：发布动态、关键评论写入。
- 世界事件：创建、结算、结束、删除写入。
- 关系变化：关系阶段变化、明显好感变化、关系摘要更新写入。
- 私聊：用户标记重要消息或片段后写入。
- 系统：今日简报写入。

### 条目字段

每条记录至少包含：

- 世界 ID。
- 发生时间。
- 类型：私聊、动态、事件、关系、主动消息、系统、手动备注。
- 关联角色，可以为空。
- 简短标题。
- 摘要正文。
- 来源对象，例如消息、动态、事件或关系变化。
- 是否可撤销影响。
- 是否可进入模型上下文。

### 模型规则

- 只读取当前世界近期、重要、未撤销记录。
- 角色相关动态/评论只在和当前角色有关时进入。
- 私聊记录只有被标记为重要时进入。
- 被隐藏、撤销或删除影响的记录不得进入上下文。

### 验收标准

- 时间线按世界隔离。
- 动态、事件、主动消息、关系变化可写入时间线。
- 时间线进入完整备份并可恢复。
- 用户可手动添加备注。
- 普通聊天不会刷屏。

## 5. 功能二：角色状态页

角色状态页回答“这个角色最近怎么样”。它不是重新写一套设定页，而是从时间线、关系、近期聊天、动态和事件中整理角色当前状态。

### 页面内容

- 当前心情或状态短句。
- 与 user 的关系阶段、好感度、关系摘要。
- 最近三件重要记忆。
- 未解决事项，例如未回应、未结算事件、关系冲突。
- 下一步倾向，例如可能主动联系、可能发动态、可能保持沉默。
- 最近活跃来源：聊天、动态、事件、主动消息。

### 交互入口

- 聊天页头像或角色资料入口可打开状态页。
- 联系人列表可显示简短状态，例如“刚发了动态”“有事件未结算”。

### 数据规则

- 状态页优先读取已有数据，不强制每次打开调用模型。
- 可提供“刷新状态摘要”按钮，由模型压缩时间线和关系信息。
- 刷新结果应保存，并进入备份。

### 验收标准

- 每个角色可打开状态页。
- 状态按世界隔离。
- 删除角色不会破坏历史时间线展示。
- 状态摘要可备份恢复。
- 没有足够信息时显示克制空状态，不胡编。

## 6. 功能三：主动消息理由可见

主动消息生成后，用户应能看到它为什么发生。理由是沉浸感的一部分，也是调试频率和关系逻辑的入口。

### 可见理由示例

- “因为你已经一整天没有回复她。”
- “因为昨晚的事件让她有点在意。”
- “因为关系进入 close，她更愿意主动找你。”
- “原本在安静时段延后，现在重新尝试。”

### 实现要求

- 主动消息写入时保存触发原因。
- 理由显示在消息详情、时间线条目或角色状态页中。
- 理由必须是用户可读文本，不展示内部枚举或预算细节。
- 若生成失败、预算不足、安静时段跳过，只在今日简报或运行记录中轻量显示。

### 验收标准

- 新主动消息有可读理由。
- 理由进入时间线。
- 理由随备份恢复。
- 用户能区分“角色主动联系”和“普通模型回复”。

## 7. 功能四：今日简报

今日简报是用户每天打开应用时的低打扰摘要，告诉用户当前世界值得注意的变化。

### 简报内容

- 未读私聊和主动消息概览。
- 新动态与重要评论。
- 待处理或已结算事件。
- 关系变化。
- 自动消息、自动动态、自动事件被跳过或延后的重要原因。
- 今日建议关注的角色。

### 交互规则

- 每个世界每天最多生成一份主简报。
- 打开应用时显示轻量入口，不强制弹窗遮挡。
- 用户可在时间线中回看历史简报。
- 简报可以先用规则生成，后续再加入模型润色。

### 验收标准

- 当天有变化时生成简报。
- 无变化时显示安静空状态。
- 简报按世界隔离。
- 简报写入时间线并进入备份。
- 简报不会频繁重复弹出。

## 8. 功能五：剧情安全阀 / 影响回滚

长期档最怕一次坏输出污染关系、记忆或事件。安全阀用于撤销某次行为带来的影响，而不只是删除 UI 上的一条文本。

### 可回滚对象

- 世界事件结算造成的关系变化。
- 主动消息写入及其时间线记录。
- 动态或评论造成的时间线记录。
- 被标记为重要的聊天记忆。
- 模型刷新出的角色状态摘要。

### 回滚规则

- 回滚优先撤销“影响”，不一定删除原始内容。
- 对用户可见地标记“已撤销影响”。
- 已撤销内容不得进入模型上下文。
- 删除事件时，若事件已造成关系影响，应提示用户是否同时撤销影响。

### 数据要求

- 高影响操作保存影响记录。
- 影响记录包含目标对象、旧值、新值、时间和来源。
- 旧备份无影响记录时，不强行支持历史回滚。

### 验收标准

- 已结算事件的关系影响可撤销。
- 撤销后关系摘要和好感度恢复或追加修正记录。
- 撤销记录进入时间线。
- 撤销后模型上下文不再引用该影响。
- 备份恢复后撤销状态保持一致。

## 9. 功能六：动态订阅与可见范围

动态不应默认所有角色都同等可见。可见范围让 Tavern Social 更像真实社交网络，也为角色间互动提供边界。

### 可见范围

- 公开：当前世界角色都可见。
- 好友可见：关系达到 familiar/close/intimate 的角色可见。
- 指定角色可见：只允许选中的角色看见。
- 屏蔽角色：公开但排除指定角色。
- 仅自己：作为用户或角色的私人状态记录。

### 交互要求

- 发布动态时选择可见范围。
- 动态卡片显示简短可见范围提示。
- 生成角色评论时，只允许可见角色参与。
- 时间线可记录动态发布，但模型上下文遵守可见范围。

### 验收标准

- 动态保存可见范围。
- 不可见角色不会生成评论。
- 切换世界后可见范围不串世界。
- 旧动态默认公开。
- 可见范围进入备份。

## 10. 功能七：角色之间的私下互动

角色之间也应能发生事，而不是所有关系都围绕 user。第一版重点做可控、低频、可解释的互动。

### 互动形式

- 角色评论其他角色动态。
- 多角色共同参与世界事件。
- 某个角色在私聊中提到另一个角色。
- 时间线记录“角色之间发生的事”。
- 后续可扩展角色间关系，但第一版不强制做复杂关系网。

### 生成规则

- 只在同一世界内发生。
- 优先选择近期有共同事件、动态可见、设定有关联的角色。
- 频率受每日上限保护。
- 角色间互动应写入时间线，必要时进入相关角色状态页。
- 不要让角色凭空知道不可见动态或私聊内容。

### 验收标准

- 角色可以评论其他可见角色动态。
- 多角色事件能影响多个参与者状态。
- 角色间互动写入时间线。
- 不可见动态不会被其他角色引用。
- 互动频率可控，不刷屏。

## 11. 数据与备份要求

新增数据必须进入完整备份，并兼容旧数据恢复。

### 新增数据概念

- 世界时间线条目。
- 角色状态摘要。
- 主动消息理由。
- 今日简报记录。
- 影响记录 / 回滚状态。
- 动态可见范围。
- 角色间互动记录。

### 兼容规则

- 旧备份没有字段时补默认值。
- 旧动态默认公开。
- 旧主动消息没有理由时显示“来自旧版本主动消息”或不显示理由。
- 旧事件没有影响记录时，不支持精确回滚，只允许删除或手动修正。

## 12. 模型上下文要求

模型上下文要更稳定，但不能无限膨胀。

### 上下文来源优先级

1. 当前角色设定和世界书。
2. 当前私聊近期消息。
3. 当前角色关系状态。
4. 当前世界近期未撤销时间线。
5. 当前角色状态摘要。
6. 相关动态、评论和事件。

### 限制

- 不跨世界读取。
- 不读取不可见动态。
- 不读取已撤销影响。
- 不把今日简报当作绝对事实；它是压缩摘要。
- 不把失败日志、预算日志、内部调度细节交给模型。

## 13. UI 与交互要求

新增 UI 必须保持通讯工具气质，避免后台化。

### 入口建议

- 底部/侧边主导航加入“时间线”或在“事件”附近加入世界记录入口。
- 角色状态页从头像、联系人详情、聊天头部进入。
- 今日简报出现在消息页顶部或时间线顶部。
- 主动消息理由出现在消息详情和时间线中。
- 回滚入口放在时间线条目详情、事件结算详情、关系变化详情中。
- 动态可见范围放在发布器中，使用下拉或分段选择。

### 移动端要求

- 不新增复杂多栏。
- 时间线、状态页、简报都应是单列可滚动。
- 重要操作按钮触控目标不小于 44px。

## 14. 分阶段实施路线

### 第一阶段：数据底座

- 增加时间线数据结构、备份兼容、基础 UI。
- 接入动态、事件、主动消息、关系变化写入时间线。
- 增加手动剧情备注。

### 第二阶段：角色状态与简报

- 增加角色状态页。
- 增加今日简报。
- 将时间线摘要接入模型上下文。

### 第三阶段：解释与安全

- 主动消息理由可见。
- 高影响行为保存影响记录。
- 支持事件结算和关系变化回滚。

### 第四阶段：社交边界

- 动态可见范围。
- 评论生成遵守可见范围。
- 旧动态迁移为公开。

### 第五阶段：角色间互动

- 支持角色评论角色动态。
- 支持多角色事件更明确地进入状态页和时间线。
- 为后续角色间关系网预留数据空间，但第一版不展开复杂关系图。

## 15. 验收标准

### 全局验收

- 新数据能保存、刷新后仍在。
- 完整备份能导出恢复。
- 旧备份能恢复，不报错，不丢旧数据。
- 所有新功能按世界隔离。
- 已撤销内容不再进入模型上下文。
- UI 在桌面和移动端都能完成主要操作。

### 功能验收

- 时间线能显示动态、事件、主动消息、关系变化和手动备注。
- 角色状态页能展示近期状态、记忆和未解决事项。
- 主动消息能展示可读理由。
- 今日简报能汇总当天重要变化。
- 事件或关系影响可撤销。
- 动态可见范围能限制角色评论和模型引用。
- 角色之间能产生低频、可解释的互动。
